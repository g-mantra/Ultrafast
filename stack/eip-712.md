# EIP-712 Typed Structured Data Signing

## Part 1: How UltraFast Uses EIP-712 and Why

UltraFast's derivatives workload exposes a **gasless turing-incomplete order lane** (§6.5) that accepts a fixed verb set - `place`, `cancel`, `modify`, `batch-cancel`, `liquidate` - outside the EVM gas mechanism. Each gasless transaction is authenticated by an **EIP-712 signature against the account's keyset**, replay-protected by a **per-account monotonic nonce**, and sequenced through the MCP pslice path at v1.1. Liquidations are emitted as protocol-signed gasless transactions by the liquidation engine itself rather than by a user.

The choice is driven by ecosystem fit rather than novelty. EIP-712 is the de facto industry standard for off-chain order authentication on EVM-compatible venues. Every standard wallet that a derivatives user is plausibly already running - MetaMask, Rabby, Frame, Trezor, Ledger, Safe (via ERC-1271), and the major mobile wallets - supports `eth_signTypedData_v4` natively. Adopting EIP-712 lets a trader sign UltraFast orders from existing tooling without bespoke key management. The wallet can then render the order parameters in a structured, human-readable prompt (market, side, size, price, expiry, nonce) instead of a raw hash.

The same pattern is used by every comparable venue. **0x Protocol** has signed limit orders with EIP-712 since v2 in 2018 - the EIP itself was authored by 0x engineers. **OpenSea Seaport** uses EIP-712 for the listing/offer payload that authorises asset spending. **dYdX v3/v4**, **CowSwap**, **LooksRare**, and **Hyperliquid** all sign orders and meta-transactions with EIP-712. UltraFast's usage is textbook for this category. The design uses a typed `Order` struct (with `market`, `side`, `size`, `price`, `expiry`, `nonce`, etc.) and a domain separator binding the signature to the UltraFast chain and to the matching engine's verifying address. The sequencer verifies the signature before the order enters the FBA tick.

Two clarifications about scope. First, the **gasless property is not from EIP-712**. EIP-712 is a signing format; it does not by itself eliminate gas. UltraFast's gasless property comes from the protocol-level decision (§6.5) to process order verbs as typed system operations rather than as EVM calls. The governance-set trading fee is charged at fill time instead of per-transaction gas. EIP-712 is doing one job: authenticating the order. Second, UltraFast is **pre-implementation**. The schema for the order struct, the verifying-contract address, and the **gasless-lane DoS budget** are open decisions in §16. The DoS budget is the per-account rate bound that prevents unfunded order spam, since the lane has no per-transaction gas cost. The working assumption is a per-account credit budget burned on cancel-without-fill and refilled on fill, plus a global per-account hard cap per tick. EIP-712 is upstream of that decision - it authenticates the signature, the DoS budget governs how many authenticated signatures the chain will admit.

## Part 2: Deep Research on EIP-712

### Specification and Status

**EIP-712** ("Typed structured data hashing and signing") was created on **September 12, 2017** by **Remco Bloemen** (@Recmo), **Leonid Logvinov** (@LogvinovLeon), and **Jacob Evans** (@dekz), engineers then working on the 0x Protocol. The proposal moved through the EIP process and reached **Final** status; it is one of the most widely cited and implemented standards in the Ethereum ecosystem. The standard solves a concrete problem that existed before it. Both `eth_sign` and `personal_sign` (`\x19Ethereum Signed Message:\n` prefix) ask the user to sign an opaque bytestring or hash. That produces both poor UX and a real phishing surface: a malicious dapp can ask the user to sign a hash that is in fact the digest of an authorising transaction.

### How It Works

EIP-712 computes a signed digest as:

```
digest = keccak256("\x19\x01" || domainSeparator || hashStruct(message))
```

The components:

- **`"\x19\x01"`** - a fixed prefix (EIP-191 version byte `0x01`) that namespaces EIP-712 signatures away from `personal_sign` and away from raw transaction payloads. A signature produced under this prefix cannot collide with an Ethereum transaction signature, which is the root of the standard's security argument.
- **`domainSeparator`** - `hashStruct(EIP712Domain)` over the `EIP712Domain` struct. The struct carries `name` (dapp/protocol name), `version` (protocol version), `chainId` (binds the signature to a specific chain, which is critical for replay protection across forks and L2s), `verifyingContract` (binds the signature to a specific contract/verifier address), and an optional `salt`. Any subset of these fields may be used; the type definition reflects which are present.
- **`hashStruct(message)`** - `keccak256(typeHash || encodeData(message))`, where `typeHash` is `keccak256` of the canonical type-string encoding (e.g. `Order(address market,uint256 size,uint256 price,uint64 expiry,uint64 nonce)`), and `encodeData` packs each field according to its type. Nested structs are hashed recursively; dynamic types (`bytes`, `string`) are pre-hashed.

The signature itself is a standard secp256k1 signature `(v, r, s)` over the digest, recoverable to the signer's address via `ecrecover`.

### Wallet and Tooling Support

EIP-712 is exposed via the JSON-RPC methods `eth_signTypedData_v3` and `eth_signTypedData_v4` (v4 adds support for arrays and recursive struct references). **MetaMask** has supported v3 since 2018 and v4 since 2019. **Rabby**, **Frame**, **Coinbase Wallet**, and the mobile wallets (Trust, Rainbow) all support v4. Hardware wallets - **Ledger** (with the ETH app's "Blind signing OFF" path) and **Trezor** - render typed messages on-device. Full type rendering on Ledger historically required a per-domain JSON descriptor served via the Ledger Live Manager. **Safe** supports EIP-712 signatures on the contract side via **ERC-1271** (`isValidSignature`), letting smart-contract accounts validate off-chain signatures by delegating to on-chain logic.

Library support is uniform: **ethers.js** (`Signer.signTypedData`), **viem** (`signTypedData`), **web3.js**, **ethers-rs**, **go-ethereum**'s `signer/core`, and OpenZeppelin Contracts (`EIP712` mixin) all expose the standard.

### Extensions Built on EIP-712

EIP-712 is the substrate for several widely deployed standards:

- **ERC-2612 (Permit)** - finalised March 2020. Adds a `permit(owner, spender, value, deadline, v, r, s)` function to ERC-20 tokens, letting a holder authorise a spender via an EIP-712 signature instead of an on-chain `approve` transaction. Native in USDC, DAI, USDS, and most ERC-20s deployed since 2021.
- **Permit2** - deployed by Uniswap in November 2022 at `0x000000000022D473030F116dDEE9F6B43aC78BA3` on every major EVM chain. Generalises permit to any ERC-20, not just tokens that implement ERC-2612 natively, by routing approvals through a single canonical contract. Used by Uniswap UniversalRouter, 1inch, CowSwap, and a long tail of integrators.
- **ERC-1271** - lets smart-contract wallets (Safe, Argent, account-abstraction wallets) validate EIP-712 signatures via an on-chain `isValidSignature(hash, signature)` view, bridging EIP-712 into smart-account land.
- **ERC-4337 / EIP-7702** - the account-abstraction stack uses EIP-712 for `UserOperation` signing in many implementations.

### Production Users

EIP-712 sits behind essentially every modern off-chain-signing protocol:

- **0x Protocol** - signed limit orders since v2 (2018); the EIP's authors were 0x engineers.
- **OpenSea Seaport** - listings and offers are EIP-712 signed; the typed payload outlines exactly what is spent and received, which is the standard's UX selling point.
- **CowSwap** - signed intents for batch auctions.
- **dYdX v3 (StarkEx) and v4 (Cosmos)** - order signing.
- **Hyperliquid** - EIP-712 signing for orders and agent approvals; integrators like Turnkey expose policy engines specifically for typed-data verification of Hyperliquid payloads.
- **LooksRare**, **Blur**, **Magic Eden** - NFT maker orders.
- **Aave Permit**, **Uniswap V3** (via Permit2) - gasless approvals.

### Security Caveats

Three failure modes are worth naming.

1. **`chainId` confusion.** A 2023 audit by Coinspect found that **more than 40 wallet vendors** did not warn the user when signing an EIP-712 payload whose `chainId` differed from the wallet's currently selected chain. The domain separator binds the signature to a chain. If the user signs for the wrong chain, a malicious dapp can replay the signature on a chain where it has meaning. Mitigation lives in wallet UX, not in the EIP.

2. **Text injection in the signing preview.** Coinspect also documented a UI flaw across major wallets where the rendered preview of an EIP-712 message could be made to differ from the actually-signed payload. Long field values, ambiguous Unicode, or struct fields rendered as separators could trick the user into signing a different message than the one shown. This is a wallet-rendering bug rather than a flaw in the standard, but it is the class of issue that EIP-712's "human-readable" promise has to survive.

3. **Phishing via authorised intents.** A signed Permit, Permit2, or Seaport approval can drain a wallet without any further on-chain action by the victim - the attacker just submits the signature. The wallet UX has to make the consequence visible. Modern wallets (Rabby, MetaMask's "Blockaid" integration) simulate the effect of typed-data signatures and warn on token-draining patterns; the EIP itself only provides the structure.

### Maturity

EIP-712 has been in production for **roughly eight years**, since 0x v2 in early 2018. It is implemented by every wallet, library, and smart-contract framework that touches signed off-chain data. For UltraFast it is the conservative, well-trodden choice. There is no novelty risk in the signing format itself, and the open questions (domain-separator parameters, order-struct schema, DoS budget) are integration choices rather than cryptographic ones.
