# BLS12-381 Signatures

## Part 1: UltraFast's use of BLS12-381

UltraFast invokes BLS12-381 pairing-based signatures in one specific place: the Threshold Simplex consensus dialect, as referenced in §2 (assumptions) and §5.1 of the whitepaper. The role is narrow and load-bearing.

At network genesis the validator set runs a one-time distributed key generation (DKG) to produce a shared BLS12-381 threshold secret. No individual validator learns the secret; each holds a share. Every consensus message in a Threshold Simplex view is a partial signature over the view's proposal, produced from a validator's share. Once `2f+1` partial signatures are collected, any party can recover a single threshold signature: a constant-size certificate of about 240 bytes per view, verifiable against the static aggregate public key. The certificate is what makes a view final under the safety semantics of §2.

UltraFast scales from `n = 30` at v1 to 100+ per §13.3. When the validator set changes, the shared secret survives via resharing. The aggregate public key stays the same, so every historical certificate signed under it remains valid without re-issuance.

The "why" is bandwidth and verification cost. With BLS aggregation, verification of a view certificate is `O(1)` in the validator count `n`. One pairing check against one aggregate public key, independent of whether 30, 100, or 300 validators co-signed. The alternative is Ed25519 or Schnorr multi-signatures with one signature per validator, which grows linearly: 30 to 100+ signatures per certificate, each 64 bytes, plus per-signature verification cost. For a chain that finalises every Threshold Simplex view at 100 ms block cadence (§12), `O(1)` certificate verification is what makes validator-set expansion viable rather than bottlenecked on signature checking. The 240-byte certificate also matters for the ZK light-client bridge of §10: smaller certificates mean cheaper Ethereum-side verification per UltraFast finalisation.

UltraFast's usage is a textbook BFT application of threshold BLS, the same shape as Dfinity, Aptos, and Algorand's voting layers. The implementation is specifically the Commonware `consensus::threshold_simplex` variant. That variant embeds threshold-BLS recovery natively into the consensus message flow rather than running it as an out-of-band aggregation step. Practical implications for UltraFast:

- **Not implemented yet.** UltraFast is pre-implementation. The threshold scheme is taken as given from Commonware; UltraFast is not modifying or re-proving it.
- **DKG path required.** Network bootstrap requires a working DKG ceremony at `n = 30`. Resharing is required for every validator-set change in §13.
- **Distinct from Ethereum's BLS use.** Ethereum uses BLS12-381 for individual validator signatures aggregated post-hoc across the entire validator set (millions of attestations). UltraFast uses threshold BLS over a much smaller curated set, with a single shared secret, recovering one canonical signature per view.

## Part 2: BLS12-381 in depth

### Curve origin and design

BLS12-381 was designed by Sean Bowe (Electric Coin Co.) in early 2017 as part of the Zcash Sapling upgrade, replacing the earlier `alt_bn128` / BN254 curve. The trigger was the Kim–Barbulescu 2016 result on the Number Field Sieve, which under conservative assumptions reduced the security level of BN254 to roughly 100 bits. BLS12-381 was constructed to deliver a clean 128-bit security level against the same and future NFS variants while remaining pairing-friendly.

The name decomposes as follows. "BLS" is Barreto–Lynn–Scott, a 2002 family of pairing-friendly elliptic curves, unrelated to the Boneh–Lynn–Shacham signature scheme that happens to share the initials. "12" is the embedding degree (the smallest `k` such that the curve's subgroup order divides `q^k − 1`, where `q` is the base field characteristic). "381" is the bit-length of the base field prime. The curve is pairing-friendly in the strong sense: it admits an efficient bilinear map `e : G_1 × G_2 → G_T`, which is the operation BLS signatures rely on for verification.

### BLS signature scheme

The signature scheme itself is Boneh–Lynn–Shacham (2001). Public keys live in one group (G_1 or G_2), signatures in the other. Signing is `σ = sk · H(m)` where `H` hashes onto the curve. Verification checks `e(σ, g_2) =? e(H(m), pk)` (in the MinPK variant) or the symmetric version (MinSig variant).

Two size variants are standard:

- **MinPK**: public keys 48 bytes (G_1), signatures 96 bytes (G_2). Used by Ethereum's consensus layer.
- **MinSig**: public keys 96 bytes (G_2), signatures 48 bytes (G_1). Used where signature-side bandwidth dominates (e.g. Chia's plot files).

For a single signature, verify takes ~1.5–2.7 ms on a modern CPU depending on implementation and variant. Signing takes ~370 µs. Pairings dominate verify cost.

### Aggregation and threshold variants

The decisive property for blockchain consensus is aggregation. Given signatures `σ_1, …, σ_n` on the same message `m` from public keys `pk_1, …, pk_n`, the aggregate is `σ_agg = σ_1 + … + σ_n` (group addition), verified against `pk_agg = pk_1 + … + pk_n` with a single pairing check. The certificate stays one curve point regardless of `n`.

**Rogue-key attack.** Naive aggregation is unsafe: an attacker who picks `pk_n = pk_attacker − pk_1 − … − pk_{n−1}` can forge an aggregate signature without controlling the other keys. Defences:

- **Proof of possession (PoP):** each validator publishes a signature on its own public key during registration, proving possession of the secret. Used by Ethereum and most production systems.
- **Distinct messages or message-prefixing:** the BDN variant (Boneh–Drijvers–Neven, 2018) adds per-key randomness so the attack does not compose.

**Threshold BLS** (Boldyreva, 2003) extends BLS to a `(t, n)` threshold setting using Shamir secret sharing over the scalar field. Any `t+1` of `n` parties holding shares can produce a signature that verifies under a single static aggregate public key, with identical verification to the non-threshold case. Properties relevant to UltraFast:

- Non-interactive signing: each party signs locally with its share, no round-trip with peers.
- Unique signatures: the recovered signature is deterministic given the message and the threshold set.
- Constant-size output: one G_1 or G_2 element regardless of `t` or `n`.
- Original Boldyreva proof is for static adversaries only. Bacho–Loss (CCS 2022) and Crites–Komlo–Maller–Tessaro–Zhu (CRYPTO 2024) give adaptive-security proofs; the latter ("Adaptively Secure BLS Threshold Signatures from DDH and co-CDH") relies only on standard assumptions.

Security of BLS multi- and aggregate signatures reduces to co-CDH (computational co-Diffie–Hellman) in the asymmetric bilinear pairing setting, plus the random-oracle assumption on the hash-to-curve function.

### Implementations

Production-grade libraries:

- **`supranational/blst`** - C/assembly with Rust, Go, Python bindings. Commissioned by the Ethereum Foundation. NCC Group audit (Jan 2021), ongoing Galois formal verification. Last release April 2026. The performance reference implementation; used by Ethereum execution-layer EIP-2537 precompiles, most consensus clients, Filecoin, Chia, and many others.
- **`zkcrypto/bls12_381`** - pure Rust, `no_std`, constant-time. The canonical "audit-friendly" Rust implementation. Targets Rust 1.63+.
- **`arkworks-rs/algebra` (`ark-bls12-381`)** - pure Rust, oriented toward zk-SNARK construction (FFTs, polynomial commitments) rather than pure signature performance.
- **`dfinity/bls12_381`** - fork of zkcrypto adding the NI-DKG primitives the Internet Computer needs.
- **`MystenLabs/fastcrypto`** - Sui's wrapper around blst with the BLS12-381 signature surface.
- **`Chia-Network/bls-signatures`** - C++ wrapper around blst with Chia-specific PoP scheme.
- **`sigp/milagro_bls`** - historical pure-Rust implementation (Apache Milagro), superseded in practice by blst.

Standardisation:

- **`draft-irtf-cfrg-bls-signature`** (CFRG of IRTF, not IETF). Latest revision `-06`, November 2025. Defines the wire format, MinPK/MinSig variants, ciphersuite identifiers, and PoP scheme. Still a draft, but stable and treated as the de facto standard.
- **`draft-irtf-cfrg-hash-to-curve`** - companion document for the `H(m)` step. Now RFC 9380 (August 2023). BLS uses the `BLS12381G1_XMD:SHA-256_SSWU_RO_` or `BLS12381G2_XMD:SHA-256_SSWU_RO_` ciphersuite.

### Production deployment

BLS12-381 is fully mature. Deployments in production:

- **Ethereum consensus layer.** Live since Beacon Chain genesis, December 2020. Every validator signs attestations and block proposals in BLS12-381 MinPK; signatures aggregate over committees, then over epochs.
- **Ethereum execution layer.** EIP-2537 added BLS12-381 curve precompiles (G_1/G_2 addition, scalar multiplication, multi-exp, pairing check, hash-to-curve). Shipped in the Pectra hard fork on 7 May 2025 at epoch 364032. Used by staking pools, restaking systems, ZK light clients, and bridges that need to verify consensus-layer signatures from inside the EVM.
- **Filecoin.** Sector proofs and consensus signatures.
- **Chia.** Block signing, plot keys, pooling protocol.
- **Dfinity / Internet Computer.** NI-DKG + threshold BLS for subnet finality and the chain-key cryptography family (including chain-key Bitcoin and Ethereum).
- **Aptos.** `aptos_std::bls12381` module supports single, multi-, aggregate, and threshold BLS in Move.
- **Sui (Mysten Labs).** Validator authority signatures via `fastcrypto`.
- **Algorand.** State-proof signatures.
- **Polkadot.** BLS used in some validator-set primitives; sync committee–style accelerators.
- **Aleo.** Account and program-signing surface.
- **Tempo (Commonware-based chain).** Uses the same `threshold_simplex` dialect UltraFast plans to use.
- **Celestia, Cosmos SDK chains via CosmWasm precompile.** Available in `cosmwasm-crypto` for application-layer verification.
- **Antelope (EOS successor).** `AntelopeIO/bls12-381` C++ implementation for the chain's signature surface.

### Performance reference points

On a single core of a modern x86_64 CPU using `blst`:

- Signature size: 48 bytes (MinSig) or 96 bytes (MinPK).
- Public key size: 96 bytes (MinSig) or 48 bytes (MinPK).
- Sign: ~370 µs.
- Verify (single): ~1.5–2.7 ms, dominated by one pairing.
- Aggregate verify (N signatures, same message): ~2–3 ms total, essentially the cost of one pairing, regardless of N, plus N group additions.
- Aggregate verify (N signatures, distinct messages): scales linearly in N pairings, but a multi-pairing batches them.

UltraFast's certificate size of "about 240 bytes per view" claimed in §5.1 of the whitepaper is consistent with one threshold signature (48 or 96 bytes) plus a session/view identifier, beacon entropy, and metadata.

### Security caveats and open considerations

- **Adaptive security.** Boldyreva's original threshold construction proves static-adversary security only. UltraFast's adversary model in §2 specifies static corruption, which matches. Migrating to adaptive security would require switching to the Bacho–Loss or Crites et al. constructions.
- **DKG is the hard part.** The BLS signature itself is well-understood; the DKG that bootstraps the shared secret is where most production bugs and security incidents live. UltraFast inherits Commonware's DKG implementation.
- **Resharing.** Validator-set churn requires verifiable secret resharing. Standard constructions exist (Desmedt–Jajodia, Wong–Wing–Wing) but add complexity beyond a one-shot DKG.
- **Post-quantum.** BLS12-381 is broken by Shor's algorithm on a sufficiently large quantum computer. Same status as every other curve in production. Not a near-term concern; a long-term migration item.
- **Subgroup checks.** Naive verification that skips the subgroup check on incoming points is exploitable (small-subgroup / cofactor attacks). All production libraries do the check by default; UltraFast inherits this discipline by using Commonware which uses blst.

### Bottom line for UltraFast

The cryptography UltraFast depends on for Threshold Simplex is the most battle-tested pairing-based signature scheme deployed in production blockchain systems. Ethereum's consensus layer has been signing with BLS12-381 continuously for over five years across hundreds of thousands of validators. The implementation UltraFast will use (`blst`, via Commonware) is the same library Ethereum, Filecoin, Chia, Sui, Aptos and others run on. The novelty in UltraFast is not in the signature scheme. It is in how the threshold variant is integrated into the Threshold Simplex consensus message flow, and that integration is Commonware's, not UltraFast's. From a cryptographic-risk standpoint, BLS12-381 is among the lowest-risk components of the UltraFast stack.
